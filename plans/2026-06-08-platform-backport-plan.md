# Port platform improvements & generic components into Forge

## Context

`platform` (`/home/c4/platform`) was **originally generated by `forge` v1.1.0** (it still carries a `forge.toml`), then hardened and extended in production. Its backend `weld-*` SDKs (`/home/c4/platform/sdks/`) and its Vue web app (`/home/c4/platform/apps/web/`) are therefore the *matured descendants* of forge's own templates. forge has since grown ~24 feature modules that already parallel almost every platform subsystem (`connectors`, `events`, `observability`, `streaming`, `middleware`, `multitenancy`, `auth`, `reliability`, `ai_chat`, …).

So this is **not a greenfield port** — it is a **divergence / back-port exercise**. Three pairwise gap analyses (backend SDKs, auth/multitenancy, frontend/agentic-UI) found that:

- Most subsystems are **at parity** — and forge is frequently the *cleaner, more generic* side (parameterized RLS GUC, pluggable outbox sink, weld-free decoupling). Those need **no action**.
- A bounded set of platform implementations have **pulled ahead** and should be back-ported as **improvements** to existing forge features.
- A handful of capabilities are **genuinely new** and worth adding as new generic components/features.
- **Auth/multitenancy is largely already merged** into forge `origin/main` via **#170 (`9051f64`)** — do **not** re-port those; only residual gaps remain.
- One finding is a **latent bug in forge today** (refined by codex, see #1): the generated gatekeeper's `/callback` silently drops `tenant_id` because of three coupled realm defects — the `tenant_id` mapper uses the wrong claim namespace (`https://forge/...` vs the gatekeeper's expected `https://platform/...`), the realm has no `userProfile` block to persist the attribute, and the gatekeeper service account lacks the `manage-users` role the callback needs. This breaks the `tenant_resolution=token_claim` RLS path end-to-end and is the single highest value/effort item.

**Goal:** deliver (a) the full prioritized back-port/improvement roadmap across all three domains, and (b) a concrete, codex-validated **first tranche of balanced quick wins** to execute now, while scoping the large new subsystems as named later phases. Per the user's choices, **all big-ticket subsystems are in scope** (cataloged with enough detail to execute), and the first tranche is **balanced quick wins** across backend + auth + frontend.

## Base, branching & conventions

- **Canonical forge:** `/workspace/c4/forge`. Branch off **`origin/main @ 9051f64`** (post-#170). Local `main` is one merge behind (`54f1ce4`, #169) — base on `origin/main`, not local `main`. The only dirty file is `README.md`; stash/ignore it.
- **Suggested branch:** `feat/platform-backport-t1` for tranche 1; later phases get their own branches.
- **Platform = read-only reference** (`/home/c4/platform`). Never modify it.
- **Respect forge's seams** (do not fork generator logic):
  - Feature: `forge/features/<name>/{feature.toml,options.py,fragments.py}` + `templates/<fragment>/<lang>/{files/…,inject.yaml}`. `inject.yaml` splices at `FORGE:*` markers (DI providers, middleware registration, settings fields).
  - Component-feature (pure UI): `feature.toml` `category="component"`, files under `templates/component_<Name>/all/files/src/...` (examples: `forge/features/stat_card/`, `forge/features/entity_list/`).
  - Backend app-template variant: `forge/templates/services/<lang>/<variant>/template.toml` (`base="services/<lang>-service-template"` for two-stage overlay; discovered by `forge/backend_app_templates.py`). Example: the existing `api-gateway` variant.
  - Platform preset: `forge/templates/platforms/<name>/platform.toml` (discovered by `forge/platform_templates.py`).
  - forge-core SDK = plain edits under `forge/templates/services/python-service-template/template/sdks/forge-core/src/forge_core/`.
- **Conventions to honor every change:** add/extend `tests/test_features_*.py`; regenerate **golden snapshots** (`tests/golden/snapshots/`) — *a new option that is off-by-default must keep all 6 goldens byte-identical*; add **matrix** scenarios (`tests/matrix/scenarios.yaml`); keep **auth SDK parity** (`tests/contract/auth_sdk_parity/`); add **e2e docker** coverage where a generated project must boot. Conventional Commits, no AI-coauthor trailers.
- **3-framework parity policy:** generic composables/components may ship **Vue-first** (forge features can be framework-scoped) with Svelte/Flutter backlogged; items that touch the existing 3-framework surface (agentic UI via `canvas-core`/`canvas-svelte`/dart packages; settings store; session-timeout fragment) are expected to ship **all three**.
