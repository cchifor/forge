# RFC-012 — ForgeQueue port

| Field | Value |
| --- | --- |
| Status | Accepted |
| Author | v2 Theme 7 |
| Created | 2026-05-17 |
| Updated | 2026-05-17 |
| Target | 1.1.0-alpha.3 |
| Supersedes | partially refines RFC-006 §"Rust queue adapters" |

## Summary

Define a single, cross-language **ForgeQueue port** — the canonical
abstract surface that every generated backend's queue scaffolding
targets. Python already ships the reference port + Taskiq/Redis/SQS
adapters under `forge/features/async_work/templates/queue_port/python/`;
this RFC codifies the operations, serialization contract, and
delivery semantics so siblings under `queue_port/node/` (BullMQ) and
`queue_port/rust/` (Apalis) can land without re-litigating shape.

The goal: swapping the queue backend within a language
(Taskiq → SQS in Python, BullMQ → Cloud Tasks in Node, Apalis → SQS
in Rust) only requires writing a new adapter. App code keeps importing
the port and stays unchanged.

## Motivation

Pre-RFC-012, only the Python port existed. Node and Rust services
generated by forge shipped queue helpers (BullMQ + ioredis under
`background_tasks/node/`, Apalis under `background_tasks/rust/`)
that leaked adapter details into the rest of the generated app:
`worker.ts` directly typed against `bullmq.Worker`, `worker.rs`
hard-coded `apalis::WorkerBuilder`. A user wanting to move Node off
BullMQ — say, to Cloud Tasks or NATS — would have to rewrite every
enqueue + handler site.

The Python side has been spared this because the `QueuePort` Protocol
gave us a stable seam. RFC-006 marked `queue_port` as a **tier-2
migration target** (Python-only, with Rust pending). RFC-012 is the
spec the tier-1 promotion depends on.

## The canonical operation set

The port covers the 80% queue use case across providers. Anything
more exotic (priority queues, fan-out, partition keys) stays inside
adapters and is opt-in via adapter-specific extension methods.

| Operation | Signature (language-neutral) | Purpose |
| --- | --- | --- |
| `enqueue` | `(topic: str, body: dict, delay_seconds: int = 0) -> message_id: str` | Submit a single message. `delay_seconds=0` is immediate delivery. |
| `consume` | `(topic: str, batch_size: int = 1) -> AsyncIterator<QueueMessage>` | Open a consumer loop on `topic`; yield messages. |
| `ack` | `(topic: str, receipt: str) -> None` | Acknowledge a message — remove it from the queue. |
| `nack` | `(topic: str, receipt: str, requeue: bool = True) -> None` | Reject a message — requeue (default) or send to DLQ if `requeue=False`. |

`QueueMessage` carries three fields: `id` (provider-assigned), `body`
(decoded JSON envelope), and `receipt` (opaque provider handle used
for `ack`/`nack`).

Schedule-at semantics: `delay_seconds` is the only scheduling knob.
Adapters that support absolute scheduling (SQS `DelaySeconds`,
BullMQ `delay`, Apalis `schedule_in`) implement it directly; adapters
that don't (Redis-list) fall back to a delayed sorted set.

Operations explicitly **out of scope** for the port:

- Priority queues — provider-specific.
- Topic fan-out / pub-sub — different domain shape; future RFC.
- Bulk enqueue — the Python port ships `enqueue` singular; if bulk
  becomes hot, it joins the spec via additive `enqueue_many` rather
  than changing `enqueue`.
- Dead-letter routing — `nack(requeue=False)` is the signal; the
  adapter wires DLQ via provider config (SQS redrive policy, BullMQ
  `failedJobsAge`, Apalis retry policy).

## Serialization contract

Every message is a JSON envelope:

```json
{
  "id": "<provider-assigned-uuid-or-message-id>",
  "body": <opaque JSON object>
}
```

- **Encoding**: UTF-8 JSON. Bytes-with-content-type messages are not
  supported by the port — wrap them in `body.data` as a base64
  string if you need to ship binary.
- **`id` field**: assigned by the producer at enqueue time (UUIDv4
  for Redis, SQS `MessageId` for SQS, BullMQ job id for BullMQ). The
  port surfaces it as `QueueMessage.id` to the consumer for
  idempotency / log-correlation.
- **`body` field**: arbitrary JSON object. Schema validation happens
  *above* the port — the application defines its own message types
  with pydantic / zod / serde and serializes them before calling
  `enqueue`.
- **Compatibility**: the envelope shape is stable across adapters.
  A Python service writing to SQS and a Rust service consuming from
  the same SQS queue both see `{"id": ..., "body": {...}}`.

## Delivery semantics

The port commits to **at-least-once delivery with explicit ack**.
Adapters that natively guarantee exactly-once (SQS FIFO with content
dedup, for instance) are free to do so under the hood; consumers
must still write idempotent handlers — that's the cross-adapter
contract.

| Concern | Port-level guarantee | Rationale |
| --- | --- | --- |
| Delivery | At-least-once | Common ground across Redis-list, SQS, BullMQ, Apalis-Redis. Strengthening to exactly-once is an adapter-level promise consumers can lean on but the port doesn't require. |
| Ordering | None | FIFO is provider-specific (SQS-FIFO, single-consumer Redis lists). Consumer code must not assume order. |
| Retry | Adapter-controlled | The port exposes `nack(requeue=True/False)`; how often the message gets re-delivered and what the back-off looks like is adapter config. Sensible defaults: 3 retries with exponential back-off; DLQ on the 4th. |
| Visibility timeout | Adapter-controlled | SQS exposes it directly; Redis-list uses RPOPLPUSH to a processing list with a separate reaper. Both are below the port. |

## Per-language realization

### Python (reference, shipping)

```python
# forge/features/async_work/templates/queue_port/python/files/src/app/ports/queue.py
class QueuePort(Protocol):
    async def enqueue(self, *, topic: str, body: dict[str, Any], delay_seconds: int = 0) -> str: ...
    async def consume(self, *, topic: str, batch_size: int = 1) -> AsyncIterator[QueueMessage]: ...
    async def ack(self, *, topic: str, receipt: str) -> None: ...
    async def nack(self, *, topic: str, receipt: str, requeue: bool = True) -> None: ...
```

Already shipped. The structural-typing `Protocol` lets adapters
(`RedisQueueAdapter`, `SqsQueueAdapter`) satisfy the port without an
explicit inheritance declaration.

### Node (this RFC)

```typescript
// services/<name>/src/app/ports/queue.ts
export interface QueueMessage {
  id: string;
  body: Record<string, unknown>;
  receipt: string;
}

export interface QueuePort {
  enqueue(args: { topic: string; body: Record<string, unknown>; delaySeconds?: number }): Promise<string>;
  consume(args: { topic: string; batchSize?: number }): AsyncIterable<QueueMessage>;
  ack(args: { topic: string; receipt: string }): Promise<void>;
  nack(args: { topic: string; receipt: string; requeue?: boolean }): Promise<void>;
}
```

Concrete adapter: `app/adapters/queue/bullmq.ts` (BullMQ + ioredis).

### Rust (this RFC)

```rust
// services/<name>/src/app/ports/queue.rs
#[async_trait::async_trait]
pub trait QueuePort: Send + Sync {
    async fn enqueue(&self, topic: &str, body: serde_json::Value, delay_seconds: u64) -> Result<String, QueueError>;
    fn consume(&self, topic: &str, batch_size: usize) -> Pin<Box<dyn Stream<Item = QueueMessage> + Send>>;
    async fn ack(&self, topic: &str, receipt: &str) -> Result<(), QueueError>;
    async fn nack(&self, topic: &str, receipt: &str, requeue: bool) -> Result<(), QueueError>;
}
```

Concrete adapter: `app/adapters/queue_apalis.rs` (Apalis + Redis).

## Adapter registration

A new option value selects which adapter ships. The existing
`queue.backend` ENUM gains two values, scoped per backend language:

| `queue.backend` | Python | Node | Rust |
| --- | --- | --- | --- |
| `none` (default) | port only, no adapter | port only, no adapter | port only, no adapter |
| `redis` | `queue_redis` | — | — |
| `sqs` | `queue_sqs` | — | — |
| `bullmq` | — | `queue_bullmq` | — |
| `apalis` | — | — | `queue_apalis` |

A user with a polyglot project chooses one queue value per project;
the resolver targets only backends whose language matches the
adapter. Picking `bullmq` in a Python-only project surfaces a hard
error ("queue_bullmq has no Python implementation") rather than
silently no-op'ing.

(Future work: per-language sub-options like `queue.python_backend`
and `queue.node_backend` if mixed-adapter projects become a common
case. The single-enum form is the pragmatic shape today.)

## Promotion to tier-1

With Node + Rust impls of `queue_port` landing in this RFC's
implementation:

1. `queue_port` covers all three built-in backends → auto-derives
   to **tier 1**. The explicit `parity_tier=2` override on the
   fragment is dropped at the same commit so the auto-derivation is
   authoritative.
2. `queue_redis` stays tier-2 (Python-only today; the Rust slot is
   filled by `queue_apalis`, which is a sibling adapter, not a
   port-of-Redis-to-Rust).
3. `queue_sqs` stays tier-2 (Python-only — RFC-006 already
   documents this as a deliberate non-promotion).
4. New `queue_bullmq` fragment is Node-only → tier-3 by
   auto-derivation. This is the correct label: BullMQ-on-Python or
   BullMQ-on-Rust is incoherent.
5. New `queue_apalis` fragment is Rust-only → tier-3 by
   auto-derivation. Same logic.

Tier-3 here means "the adapter is language-specific by design"
rather than "the feature is Python-only" — the port itself is
tier-1, which is what the cross-language parity contract cares
about.

## Drawbacks

- **One enum, five values, four "no-op" cells** in the matrix
  above. The trade-off vs. per-language sub-options: simpler
  surface area today, awkward shape if mixed-adapter projects
  proliferate. We can split later non-breakingly by adding
  `queue.python_backend` etc. and keeping `queue.backend` as a
  deprecated alias.
- **`async_trait` in the Rust port** introduces a heap allocation
  per call. Acceptable at queue throughputs (1k–10k msgs/sec); if
  it ever bites we can move to GAT-based `impl Future` returns in a
  future RFC.

## Alternatives considered

- **Per-language port specs**. Rejected: the whole point of a port
  is that the same domain shape sits behind every adapter. Letting
  Python and Rust diverge their `enqueue` signatures would mean a
  service moving an event from Python to Rust couldn't reuse the
  envelope.
- **Drop `delay_seconds` from the port**. Rejected: delayed delivery
  is on the 80% path (rate-limited webhooks, retry back-offs). Redis
  has it via sorted-set polling, SQS has it native, BullMQ has it
  native, Apalis has it native. The cost of including it is low.
- **Add a `peek` / `length` operation**. Rejected for now: the
  Python port doesn't have one, Apalis exposes it indirectly via
  storage metrics, BullMQ has `getJobCounts()`. Cross-provider
  uniformity is weak; add it if a real consumer wants it.
- **Hard-bind to a JSON Schema for the envelope**. Rejected: the
  port commits to the *outer* envelope (`{id, body}`) only. Inner
  schema is the application's concern — wrapping it in a port-level
  schema would force every project to depend on a forge-provided
  pydantic / zod / serde root type.

## Open questions

- **Tracing propagation**: should the envelope carry a
  `trace_context` field for cross-service OTLP span linking?
  Current answer: not at the port level — middleware can stuff the
  trace header into `body._otel` and the consumer adapter can lift
  it back into the active span. Revisit if a real cross-service
  trace contract emerges.
- **DLQ shape**: when `nack(requeue=False)` fires, where does the
  message land? Today: provider default (SQS redrive, BullMQ
  failed-job store, Apalis retry-exhaust). Should the port expose a
  `dlq_topic` parameter? Current answer: no — adapters wire their
  own DLQ semantics; surfacing it at the port would force every
  adapter to support DLQ uniformly, which Redis-list doesn't.
