# ADR-005: Target weld-* SDKs as the consumer surface for Python services

- Status: Accepted
- Author: forge team
- Date: 2026-05-11
- Depends on: ADR-001 (pragmatic hexagonal), ADR-002 (ports-and-adapters)

## Context

The platform completed its **2026-05 SDK restructure**: every cross-cutting concern previously living inside a per-service `src/service/` shim now ships as one of 10 `weld-*` packages under `platform/sdks/weld-*`, organized in a Tier 0 → Tier 2 acyclic dependency DAG.

| Tier | SDKs |
|---|---|
| Tier 0 (foundation, no inter-SDK deps) | `weld-core`, `weld-auth`, `weld-observability`, `weld-events`, `weld-connectors`, `weld-airlock`, `weld-mcp-template` |
| Tier 1 (one weld-* dep) | `weld-http-client` (→ weld-observability, weld-core), `weld-streaming` (→ weld-events) |
| Tier 2 (composition) | `weld-fastapi` (→ weld-core, weld-auth) |

Forge's Python service template predated this. It vendored a `src/service/` tree (~36 files: client/circuit_breaker, observability/correlation, observability/json_logging, db/aio, repository, uow, security, api/{filtering,pagination}, domain/{account,user,config}, discovery, tasks) that was a literal duplicate of what the weld-* packages now own. Every platform service (`services/airlock`, `services/knowledge`, `services/profile`, …) had migrated to the SDK imports; only forge-generated services continued to ship the shim.

## Decision

**Forge templates target weld-* SDKs as first-class dependencies for Python services.** The `src/service/` shim is removed from the base template. New Python services generated by forge import directly from `weld.*` and consume the SDKs via monorepo path deps (`pyproject [tool.uv.sources]` → `../../sdks/weld-*`, Dockerfile `additional_contexts: { sdks: ./sdks }`).

### Consumption modes

Driven by the new `sdk_consumption` copier prompt:

| Mode | When to pick |
|---|---|
| `monorepo` (default) | The generated service lives under `platform/services/<name>/`. Path deps to `../../sdks/weld-*`. |
| `standalone` | A future state once weld-* publishes to PyPI. Currently emits a warning at generation time. |
| `none` | Preserves the legacy 1.1.x behavior and keeps `src/service/`. Existing services pin to this until they migrate. |

### Wiring strategy: feature-fragment-driven

We considered three approaches for keeping the base `src/app/` independent of `src/service/`:

1. **Jinja-conditional imports** — every `src/app/*.py` becomes `.jinja`-ified with `{% if sdk_consumption == "none" %}from service.X{% else %}from weld.X{% endif %}`. Most invasive; touches every consumer file forever.
2. **Dual subdirectory trees** — `template/` (legacy) and `template_weld/` (new); Copier `_subdirectory` picks one. Duplicates most of the file tree, divergence is inevitable.
3. **Feature-fragment-driven wiring (chosen)** — base template imports directly from `weld.*`. Optional capabilities (events, streaming, connectors, airlock, mcp_template, async tasks) live in feature fragments that `inject.yaml` their wiring into the base `src/app/`. Closest to forge's existing architecture.

Option 3 is the right shape because it aligns with how every other forge feature already works: the base template owns no infrastructure choices, and fragments layer in what's enabled. The new feature modules added in PR 1 (events, streaming, connectors, airlock, mcp_template) follow exactly this pattern.

### Out of scope

- Publishing weld-* to PyPI. The `standalone` consumption mode is reserved for when this lands.
- TypeScript/Rust weld-* SDKs. The Node and Rust templates align on conventions (Dockerfile shape, compose fragment, Traefik labels, `/api/v1/health/live`, non-root runtime) but keep their native library choices (Fastify + Prisma, Axum + sqlx).
- Migrating existing forge-generated services. Users who haven't yet adopted weld-* either stay on 1.1.x or regenerate with `sdk_consumption=none` until they're ready.

## Consequences

**Pro.** Generated services drop straight into `platform/services/` and match every existing service's import surface. Forge no longer ships duplicate code that's already maintained inside the platform monorepo. Generator output is ~2000 fewer lines per service.

**Pro.** Cross-cutting bug fixes (auth, observability, repository internals) land once in weld-* and reach every generated service on the next dependency update — no template regeneration required.

**Con.** Forge is now coupled to the platform's SDK release cadence. Breaking changes in weld-* propagate to forge-generated services on their next `uv sync`. This is the correct trade — the alternative is dual maintenance — but it means forge's release notes need to call out weld-* version pins explicitly.

**Con.** Standalone consumption is currently unavailable. Until weld-* is on PyPI, forge-generated services that aren't part of the platform monorepo have to pick `sdk_consumption=none` and accept the legacy shim. This will be revisited when weld-* ships its first PyPI release.
